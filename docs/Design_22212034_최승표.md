## 1. Intorduction
본 문서에서는 통합 관리 프로그램인 'Reserversity' 프로젝트의 소프트웨어 설계 명세서이며, 시스템을 "어떻게" 구현하고자 하는지를 담고있습니다.

1.1. 목적(Purpose)
- 본 문서는 영남대학교 'Reserversity'의 비즈니스 요구사항을 정밀하게 분석하고, 이를 안정적이고 확장 가능한 백엔드 시스템으로 구축하기 위한 기술적 토대를 마련하는 것을 목적으로 한다. 
- 사용자 인터페이스(Actor)와 시스템 내부 로직 간의 상호작용을 명확히 정의하고 컴포넌트 간의 결합도를 낮추어, 개발 구현 단계에서 발생할 수 있는 설계 오류와 의존성 충돌을 최소화하고자 한다.
- 자원 속성의 가변성을 수용하기 위한 합성 구조 설계와, 예약 자격 검증 로직의 파편화를 막기 위한 절략 패턴의 구조적 배치를 명시하여 시스템의 높은 응집도와 유지보수성을 달성함을 목적으로 한다.

1.2. 범위(Scope)
- 본 설계의 범위는 'Reserversity' 시스템의 프론트엔드 사용자 인터페이스 설계와 백엔드 핵심 비즈니스 로직 설계를 모두 포괄하며, 구체적인 범위는 다음과 같다
- UI설계: 사용자 경험을 고려한 화면 프로토타입 및 애플맄이션 상호작용 흐름 정의
- 백엔드 도메인 아키첵처 설계: 객체지향 설계 원칙을 적용한 예약 제어, 동적 메타데이터 처리, 유효성 검증 정책 클래스 구조 정의
- 데이터 모델링: 통합 자원 관리를 위한 영속성 데이터베이스의 논리적/물리적 구조 정의

| 클래스명 | Role | Attributes | Operations |
| :--- | :--- | :--- | :--- |
| **ResourceAdminUI** | 관리자가 새로운 자원을 등록하고 상태를 변경하는 인터페이스 | **`- inputResourceName: String`**<br>새로 등록할 자원의 이름 입력값<br><br>**`- selectedStatus: String`**<br>자원의 현재 대여 가능 상태 선택값 | **`+ renderResourceDashboard(): void`**<br>자원 목록과 상태를 한눈에 보는 대시보드를 렌더링한다.<br><br>**`+ submitNewResource(): void`**<br>입력된 정보를 바탕으로 새로운 자원 등록을 요청한다.<br><br>**`+ changeResourceStatus(): void`**<br>특정 자원의 물리적 상태(예: 고장, 정상) 변경을 요청한다. |

| **ResourceAdminService** | 카테고리 및 자원 생성, 자원의 상태 변경 흐름 제어 | **`- categoryRepository: CategoryRepository`**<br>자원 생성 시 카테고리 유무를 확인하기 위한 저장소 참조 | **`+ addCategory(req: CategoryDto): Long`**<br>새로운 카테고리를 생성하고 저장된 ID를 반환한다.<br><br>**`+ addResource(req: ResourceDto): Long`**<br>지정된 카테고리 하위에 새로운 자원을 생성하고 ID를 반환한다.<br><br>**`+ updateStatus(resourceId: Long, status: ResourceStatus): void`**<br>특정 자원의 대여 상태를 변경하고 DB에 반영한다. |
| **Resource** | 자원의 식별자, 상태, 가변 속성 및 소속 카테고리를 아우르는 엔티티 | **`- id: Long`**<br>자원의 고유 식별자 (PK)<br><br>**`- name: String`**<br>자원의 명칭 (예: 스터디룸 A)<br><br>**`- status: String = "AVAILABLE"`**<br>현재 자원의 물리적 대여 가능 상태<br><br>**`- category: Category`**<br>이 자원이 속해 있는 상위 카테고리 참조<br><br>**`- customFields: List<CustomField>`**<br>자원별로 유연하게 추가되는 특수 메타데이터 리스트 | **`+ isAvailable(): boolean`**<br>자원 스스로 현재 대여 가능한 상태인지 판별한다.<br><br>**`+ addCustomField(field: CustomField): void`**<br>자원에 새로운 속성(키-값)을 추가한다.<br><br>**`+ removeCustomField(key: String): void`**<br>자원에서 특정 메타데이터 속성을 제거한다. |

| **CustomField**<br>`<<ValueType>>` | 자원별로 추가되는 특수 메타데이터 (키-값 쌍) | **`+ key: String`**<br>속성의 이름 (예: "OS", "수용인원")<br><br>**`+ value: String`**<br>속성의 실제 값 (예: "Windows 11", "6명") | *(없음)* |
| **ResourceRepository**<br>`<<Interface>>` | 특정 자원을 데이터베이스에서 조회하는 영구 저장소 | *(없음)* | **`+ findById(id: Long): Optional<Resource>`**<br>ID를 조건으로 자원 엔티티를 조회한다. |

### 3. Reservation Domain (예약 핵심 도메인)
* **도메인 목적:** 사용자가 자원을 특정 시간대에 점유하는 행위와 검증 정책을 총괄.

| 클래스명 | Role | Attributes | Operations |
| :--- | :--- | :--- | :--- |
| **ReservationUI** | 사용자가 자원과 시간을 선택하여 예약을 요청/취소하는 인터페이스 | **`- selectedResourceID: Long`**<br>사용자가 예약을 위해 선택한 대상 자원의 ID<br><br>**`- selectedPeriod: ReservationPeriod`**<br>사용자가 화면에서 지정한 예약 희망 시간대 | **`+ renderResourceList(): void`**<br>예약 가능한 자원 목록을 화면에 렌더링한다.<br><br>**`+ requestReservation(): void`**<br>선택된 자원과 시간 정보로 서버에 예약 승인을 요청한다.<br><br>**`+ requestCancel(reservationID: Long): void`**<br>확정되었던 특정 예약을 취소 요청한다. |
| **ReservationService** | 정책들을 순회하며 예약을 검증하고 DB 트랜잭션을 관리하는 제어자 | **`- policies: List<EligibilityPolicy>`**<br>예약을 검증할 때 사용할 전략(정책) 인터페이스의 모음 | **`+ register(req: ReservationRequestDto): ReservationResponseDto`**<br>정책 검증 후 예약을 확정하여 저장한다.<br><br>**`+ cancel(reservationId: Long): void`**<br>특정 예약을 찾아 상태를 취소로 변경한다.<br><br>**`+ approveForcefully(reservationId: Long): void`**<br>관리자 권한으로 예외 상황의 예약을 강제 승인한다. |
| **EligibilityPolicy**<br>`<<Interface>>` | 예약 신청이 시스템의 규칙에 부합하는지 검증하는 전략 패턴의 추상체 | *(없음)* | **`+ check(user: User, resource: Resource, period: ReservationPeriod): void`**<br>규칙 위반 시 예외를 발생시키며, 통과 시 아무 동작도 하지 않는다. |
| **TimeConflictPolicy** | 신청한 시간대에 이미 해당 자원이 점유되어 있는지 검증하는 구체 정책 | **`- reservationRepository: ReservationRepository`**<br>기존 예약 내역을 조회하기 위한 저장소 참조 | **`+ check(user: User, resource: Resource, period: ReservationPeriod): void`**<br>해당 자원의 지정 시간대 중복 예약 유무를 판별한다. |
| **PenaltyPolicy** | 예약 신청자가 현재 제재 상태인지 검증하는 구체 정책 | *(없음)* | **`+ check(user: User, resource: Resource, period: ReservationPeriod): void`**<br>사용자 객체의 제재 정보를 확인하여 예약 자격을 판별한다. |
| **Reservation** | 특정 사용자가 특정 자원을 점유하고, 관리자 승인 이력을 함께 갖는 핵심 엔티티 | **`- id: Long`**<br>예약 데이터의 고유 식별자 (PK)<br><br>**`- resource: Resource`**<br>이 예약이 점유하고 있는 대상 자원<br><br>**`- user: User`**<br>예약을 신청한 사용자(예약자)<br><br>**`- approver: User = null`**<br>예약을 승인한 관리자 사용자. 승인 전에는 null일 수 있다.<br><br>**`- period: ReservationPeriod`**<br>예약이 유지되는 구체적인 시간대 정보<br><br>**`- status: String = "CONFIRMED"`**<br>예약의 현재 처리 상태 | **`+ create(user: User, resource: Resource, period: ReservationPeriod): Reservation`**<br>예약자와 자원, 시간 정보를 바탕으로 새로운 예약 객체를 안전하게 조립하여 반환한다.<br><br>**`+ approve(approver: User): void`**<br>예약의 승인자를 기록하고 승인 상태로 전환한다.<br><br>**`+ cancel(): void`**<br>예약의 상태를 취소 상태로 변경한다.<br><br>**`+ forceApprove(): void`**<br>예약의 상태를 관리자 직권으로 강제 승인 상태로 변경한다. |
| **ReservationPeriod**<br>`<<ValueType>>` | 시작과 종료 일시를 하나로 묶어 관리하며 시간 중복을 계산하는 값 객체 | **`- start: LocalDateTime`**<br>대여가 시작되는 시간<br><br>**`- end: LocalDateTime`**<br>대여가 종료되는 시간 | **`+ isOverlapping(other: ReservationPeriod): boolean`**<br>다른 시간대 객체와 비교하여 1분이라도 겹치는 구간이 있는지 수학적으로 계산한다. |
| **ReservationRepository**<br>`<<Interface>>` | 예약 데이터를 DB에 저장/조회하고 시간 중복을 쿼리하는 영구 저장소 | *(없음)* | **`+ hasOverlap(resourceId: Long, period: ReservationPeriod): boolean`**<br>특정 자원의 특정 시간대에 겹치는 기존 예약이 존재하는지 DB 레벨에서 확인한다.<br><br>**`+ save(reservation: Reservation): Reservation`**<br>검증이 완료된 예약 객체를 DB에 영구 저장한다.<br><br>**`+ findById(id: Long): Optional<Reservation>`**<br>ID를 조건으로 예약 엔티티를 조회한다. |


## 3. Sequence diagram
### 3.1 카테고리 추가
<img src="./img/diagram/SequenceDiagram/01카테고리추가.svg">

카테고리 추가는 관리자가 모달에서 입력한 이름과 설명을 서버로 전송하고, 서비스가 이를 저장한 뒤 생성 결과를 돌려주는 흐름이다.
중복 검증과 저장 책임이 분리되어 있어, UI와 도메인 로직이 자연스럽게 분리된다.

### 3.2 카테고리 수정
<img src="./img/diagram/SequenceDiagram/02카테고리수정.svg">

카테고리 수정은 기존 카테고리의 이름 또는 설명을 변경하는 흐름이다.

### 3.3 카테고리 삭제
<img src="./img/diagram/SequenceDiagram/03카테고리삭제.svg">

카테고리 삭제는 선택된 카테고리를 찾아 제거하는 과정으로, 삭제 전에는 해당 카테고리의 사용 여부를 함께 고려해야 한다.
삭제 결과는 관리 화면으로 다시 전달되어 목록 갱신에 사용된다.

---

### 3.4 카테고리 조회
<img src="./img/diagram/SequenceDiagram/04카테고리조회.svg">

카테고리 조회는 관리자가 등록된 카테고리 목록을 확인하는 흐름이다.
목록 조회 결과를 화면에 렌더링하여 이후 등록이나 수정 작업의 기준 정보로 활용한다.

---

### 3.5 신규 자원 등록하기
<img src="./img/diagram/SequenceDiagram/05신규자원등록.svg">

신규 자원 등록은 자원 정보와 카테고리, 초기 상태를 입력받아 새로운 자원을 생성하는 흐름이다.
생성된 자원은 저장 후 목록에 반영되어 이후 예약 가능 여부 판단에 사용된다.

---

### 3.6 자원 수정하기
<img src="./img/diagram/SequenceDiagram/06자원수정하기.svg">

자원 수정은 자원의 이름, 상태, 부가 속성 등을 변경하는 흐름이다.
수정된 값은 저장소에 반영되고, 관리 화면은 변경된 상태를 다시 보여준다.

---

### 3.7 자원 삭제하기
<img src="./img/diagram/SequenceDiagram/07자원삭제하기.svg">

자원 삭제는 더 이상 사용하지 않는 자원을 목록에서 제외하는 흐름이다.
삭제 시점에는 현재 예약 가능 상태나 연관 데이터가 있는지 함께 확인하는 것이 필요하다.

---

### 3.8 자원 조회하기
<img src="./img/diagram/SequenceDiagram/08자원조회하기.svg">

자원 조회는 사용자가 예약 가능한 자원 목록을 확인하는 기본 흐름이다.
조회된 자원은 이후 예약 신청의 대상이 되며, 상태 값과 함께 화면에 표시된다.

---

### 3.9 사용자 정의 컬럼 추가
<img src="./img/diagram/SequenceDiagram/09사용자정의컬럼추가하기.svg">

사용자 정의 컬럼 추가는 자원별 특수 속성(OS, 수용 인원 등)을 유연하게 등록하는 흐름이다.
고정 속성만으로 표현하기 어려운 자원 정보를 확장 가능한 형태로 관리한다.

---

### 3.10 부정사용자 이용 권한 제한하기
<img src="./img/diagram/SequenceDiagram/10부정사용자이용권한제한하기.svg">

부정사용자 제재는 관리자 요청에 따라 사용자의 예약 권한을 제한하는 흐름이다.
제재 정보는 사용자 도메인에 누적되어 이후 예약 가능 여부 판단에 직접 반영된다.

---

### 3.11 제재 수위 및 사유 조정
<img src="./img/diagram/SequenceDiagram/11제재수위및사유조정.svg">

제재 수위 및 사유 조정은 기존 제재를 더 강하게 또는 약하게 수정하는 흐름이다.
운영 정책에 따라 사유와 종료 시점을 함께 조정할 수 있다.

---

### 3.12 사용자 이용 권한 복구
<img src="./img/diagram/SequenceDiagram/12사용자이용권한복구.svg">

사용자 이용 권한 복구는 제재가 해제된 사용자를 정상 상태로 되돌리는 흐름이다.
복구가 완료되면 사용자는 다시 예약 가능 대상이 된다.

---

### 3.13 예약 등록하기
<img src="./img/diagram/SequenceDiagram/13예약등록하기.svg">

예약 등록은 사용자가 선택한 자원과 시간대를 기준으로 예약을 생성하고 저장하는 핵심 흐름이다.
정책 검증과 시간 충돌 검사가 먼저 수행되고, 통과한 경우에만 예약이 확정된다.

---

### 3.14 예약 수정하기
<img src="./img/diagram/SequenceDiagram/14예약수정하기.svg">

예약 수정은 기존 예약의 시간이나 대상 자원을 변경하는 흐름이다.
수정 요청은 다시 정책 검증과 충돌 검사를 거쳐야 하므로 신규 등록과 유사한 검증 절차를 따른다.

---

### 3.15 예약 삭제하기
<img src="./img/diagram/SequenceDiagram/15예약삭제하기.svg">

예약 삭제는 확정된 예약을 취소하거나 완전히 제거하는 흐름이다.
삭제 후에는 자원 점유 정보가 해제되어 다른 예약이 가능해진다.

---

### 3.16 예약 조회하기
<img src="./img/diagram/SequenceDiagram/16예약조회하기.svg">

예약 조회는 사용자가 자신 또는 시스템의 예약 목록을 확인하는 흐름이다.
조회 결과는 이후 출력, 필터링, 수정, 삭제의 출발점이 된다.

---

### 3.17 사용자 제재 확인
<img src="./img/diagram/SequenceDiagram/17사용자제재확인.svg">

사용자 제재 확인은 예약 요청 전에 현재 사용자의 제재 상태를 검사하는 흐름이다.
제재가 활성 상태라면 예약 진행을 중단하고, 정상 상태일 때만 다음 검증 단계로 넘어간다.

---

### 3.18 예약 내역 출력하기
<img src="./img/diagram/SequenceDiagram/18예약내역출력하기.svg">

예약 내역 출력은 조회된 예약 데이터를 엑셀이나 PDF 같은 형식으로 내보내는 흐름이다.
기간과 출력 형식을 지정해 원하는 범위의 예약만 추출할 수 있다.

---

### 3.19 출력 조건 필터링하기
<img src="./img/diagram/SequenceDiagram/19출력조건필터링하기.svg">

출력 조건 필터링은 내보내기 전에 날짜 범위, 자원, 상태 등의 조건을 좁히는 흐름이다.
필터가 적용되면 출력 대상이 정제되어 불필요한 예약 데이터가 제외된다.

---

## 4. State machine diagram
<img src="./img/diagram/StateMachineDiagram.png">

이 상태도는 시스템의 전체 사용 흐름을 대시보드, 서비스 세션, 예약 조회, 예약 내보내기로 나누어 표현한다.
시작은 `turnOn`으로 대시보드가 렌더링되는 지점이며, 사용자가 `requestService`를 선택하면 서비스 세션으로 진입해 예약 관련 검증을 수행한다.

서비스 세션 내부에서는 먼저 사용자 검증이 실행되어 제재 여부와 예약 가능 여부를 판단한다. 이 단계에서 통과하면 자원 검증으로 이동해 시간 충돌과 자원 가용성을 확인하고, 실패하면 `반려` 상태로 빠진다.

자원 검증을 통과하면 `예약처리`로 진입하며, 이 안에서 예약확정 상태가 실제 예약 생성과 정책 검증 후 저장을 담당한다. 저장 실패가 발생하면 동일 상태로 재시도하는 구조이므로, 저장 안정성과 재처리 흐름이 드러난다.

예약이 확정된 이후에는 예약 취소 요청이나 관리자 강제 승인 요청이 별도 전이로 연결된다. 반려 상태에서도 관리자가 강제 승인을 수행할 수 있도록 분기되어 있어 예외 처리와 운영 개입 경로가 함께 표현된다.

대시보드에서는 `requestExport`를 통해 예약 조회 상태로 이동할 수 있으며, 예약 조회에서 조건을 적용한 뒤 예약 내보내기 상태로 넘어간다. 내보내기 상태는 포맷 설정과 다운로드 완료를 담당하고, 작업 종료 후 다시 대시보드로 복귀한다.

전체적으로 이 상태도는 예약 신청, 검증, 저장, 취소, 강제 승인, 조회, 출력이라는 핵심 업무 흐름을 하나의 순환 구조로 묶어 보여준다.

## 5. Implementation requirements

### 5.1. Core functional requirements
1. 시스템은 실행 즉시 로그인 없이 대시보드를 표시해야 한다.
2. 사용자는 카테고리, 자원, 예약, 사용자 제재, 내보내기 기능을 대시보드에서 직접 수행할 수 있어야 한다.
3. 예약 등록 시 시스템은 사용자 제재 여부, 자원 가용성, 시간 충돌을 순차적으로 검증해야 한다.
4. 예약 승인 시 승인자 식별자(학번 또는 사번)는 반드시 기록되어야 하며, 누가 승인했는지 추적 가능해야 한다.
5. 예약 수정 및 삭제는 기존 예약의 식별자를 기준으로 수행되어야 하며, 변경 결과는 즉시 목록에 반영되어야 한다.
6. 관리자는 사용자 제재를 등록, 수정, 해제할 수 있어야 하며 예외적으로 강제 승인을 수행할 수 있어야 한다.
7. 예약 조회 결과는 기간, 자원, 식별자 등의 조건으로 필터링할 수 있어야 한다.
8. 필터링된 예약 데이터는 Excel 또는 PDF 형식으로 내보낼 수 있어야 한다.

### 5.2. Data and domain requirements
1. 카테고리는 하나 이상의 자원을 포함할 수 있으며, 자원이 포함된 카테고리는 삭제를 제한해야 한다.
2. 자원은 하나의 카테고리에만 속해야 하며, 사용자 정의 컬럼을 통해 확장 가능한 속성을 가질 수 있어야 한다.
3. 예약은 예약자(사용자), 자원, 승인자, 시간대, 상태 정보를 반드시 포함해야 한다.
4. 제재 정보는 사용자 엔티티에 종속되어 저장되며, 만료 시 자동으로 해제 가능해야 한다.
5. 예약 상태는 적어도 확정, 취소, 강제 승인 흐름을 표현할 수 있어야 한다.
6. 예약 승인 시 승인자 식별자는 필수 기록 항목이어야 하며, 예약 승인 전 검증 대상에 포함되어야 한다.

### 5.3. Validation and exception requirements
1. 예약 요청 시 제재 대상자는 기본적으로 예약을 진행할 수 없어야 한다.
2. 동일 자원에 대해 시간이 겹치는 예약은 허용되어서는 안 된다.
3. 저장 실패가 발생한 예약은 재시도할 수 있어야 한다.
4. 예외 예약은 관리자 직권 승인 절차를 통해서만 진행되어야 한다.
5. 상태 변경과 삭제는 사용자 실수를 줄이기 위해 확인 절차를 거쳐야 한다.

### 5.4. Export requirements
1. 내보내기 기능은 예약 조회 결과를 기반으로 수행되어야 한다.
2. 사용자는 내보내기 전에 기간과 출력 형식을 선택할 수 있어야 한다.
3. 출력 형식은 최소한 Excel과 PDF를 지원해야 한다.
4. 내보내기 결과는 다운로드 가능 파일로 제공되어야 한다.

### 5.5. Non-functional requirements
1. 프로그램은 실행 직후 빠르게 사용할 수 있어야 하며, 초기 화면 진입이 지연되어서는 안 된다.
2. 예약 및 조회 화면은 사용자가 현재 상태를 직관적으로 파악할 수 있도록 구성되어야 한다.
3. 주요 동작은 예약 충돌이나 권한 문제를 즉시 식별할 수 있도록 명확한 안내를 제공해야 한다.
4. 데이터는 로컬 환경에서 안정적으로 저장 및 조회되어야 하며, 장기적으로는 백업 가능한 구조를 가져야 한다.

### 5.6. Interface requirements
1. 대시보드는 카테고리별 자원 현황과 예약 상태를 한눈에 보여주어야 한다.
2. 예약 조회 화면은 조건 필터와 출력 버튼을 제공해야 한다.
3. 내보내기 화면은 기간, 자원, 식별자, 출력 형식 선택 항목을 제공해야 한다.
4. 관리자 화면은 사용자 제재와 자원 관리 기능을 분리된 흐름으로 제공해야 한다.

## 6. Glossary

1. **Reserversity**  학과 자원 예약과 관리를 통합하는 본 프로젝트의 시스템 이름이다.

2. **대시보드 (Dashboard)**  시스템 실행 직후 가장 먼저 보이는 메인 화면으로, 자원과 예약 현황 및 주요 관리 기능의 진입점이다.

3. **카테고리 (Category)**  자원을 성격별로 묶기 위한 상위 분류 단위이다. 강의실, 기자재, 도서처럼 서로 다른 자원군을 구분하는 데 사용된다.

4. **자원 (Resource)**  예약 또는 대여의 대상이 되는 개별 항목이다. 하나의 카테고리에 속하며 상태와 사용자 정의 속성을 가질 수 있다.

5. **예약 (Reservation)**  특정 사용자가 특정 자원을 특정 시간대에 점유하도록 기록한 핵심 거래 단위이다.

6. **시간 충돌 (Time Conflict)**  동일 자원에 대해 이미 존재하는 예약 시간과 새 예약 시간이 겹치는 상태를 의미한다.

7. **제재 (Penalty)**  규정 위반 사용자에게 부여되는 예약 제한 정보이다. 제재가 활성화되면 예약 요청이 제한된다.

8. **사용자 정의 컬럼 (Metadata)**  자원별 특성 차이를 반영하기 위해 관리자가 추가하는 확장 속성이다.

9. **출력/내보내기 (Export)**  필터링된 예약 데이터를 Excel 또는 PDF 파일로 변환하여 외부 문서로 저장하는 기능이다.

10. **강제 승인 (Force Approve)**  일반적인 예약 제한이나 반려 상태를 관리자 권한으로 예외 승인하는 처리이다.

11. **예약 조회 (Reservation Query)**  특정 조건에 맞는 예약 목록을 화면이나 출력 대상으로 확인하는 행위이다.

12. **예약 처리 (Reservation Processing)**  예약 검증, 저장, 취소, 예외 승인까지 이어지는 전체 예약 제어 흐름을 의미한다.

13. **승인자 (Approver)**  예약 건을 승인한 관리자 사용자로, 예약 승인 시 반드시 기록되어야 하는 식별자이다.
